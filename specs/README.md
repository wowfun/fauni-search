# Specs Guide

本目录用于维护 FauniSearch 的规范文档，采用主题分组 + 以下结构：
- `spec.md`：长期有效的稳定规范，必需，且至少应包含：开头目的段落、`## 范围`、`## 关联主题`
- `plan.md`：可迭代的阶段实施方案，非必要不创建
- `tasks.md`：执行清单与状态跟踪，非必要不创建
- `*.md`：额外补充文件（自由命名），必须在 `spec.md` 或 `plan.md` 中有链接入口

## 术语表达
- 重要概念为避免歧义，首次出现时使用"中文 + 英文"并列（例如：本地优先 (Local-First)）
- 若英文术语已是行业标准且更清晰，可直接使用英文（例如：embedding, SQLite）
- 同一文档内概念命名应保持一致，不混用多个同义写法
- 如有必要，`spec.md` 的第一章可设为 `## 关键术语 (Terminology)`
    - 只列真正需要统一理解的关键术语，避免把对象名、章节主语或正文概念机械搬入术语表
    - 术语项可只写"中文 (English)"；如需定义，也只给最小必要的基本定义
    - 英文术语按通行写法书写
    - 术语章节不承载业务逻辑、约束条件或交互规则
    - 已在术语章节定义的术语，后续正文直接沿用，不重复解释

## 规范事实源
- 每条稳定规范只应在一个最合适的 `spec.md` 中作为事实源定义
- 其他专题应通过链接、承接说明或最小必要引用复用，不重复改写同一约束
- 若某条规则已有上游专题承载，变更时应优先更新其事实源专题，而不是在下游专题分别修改

## CHANGELOG
- 全局变更日志文件固定为 [`CHANGELOG.md`](../CHANGELOG.md)
- `CHANGELOG.md` 为低频日志，仅记录主要变更，不记录 `plan.md`、`tasks.md` 的迭代细化
- 在日期块 `## YYYY-MM-DD` 下记录（倒序排序），块内按 Added/Changed/Fixed/... 分组
- 对已有小节追加内容，不重复创建同名小节

## 编号与归档规则
- 基础主题使用 `000-099`
- 功能专题使用 `100+`
- 编号采用 `+10` 留白策略，便于后续插入相关主题
- 功能专题建议使用长期能力域命名（例如：`100-text-search`、`110-image-search`）
- 对会直接驱动代码实现的 `100+` 功能专题，`spec.md` 推荐补充 `## 功能需求` 与 `## 验收标准`
- 功能需求应优先描述黑盒行为与系统外显结果，可使用 `FR-*` 编号
- 验收标准应优先描述可验证结果与验收场景，可使用 `AC-*` 编号；可以接近 TDD 的 Given/When/Then，但不承载测试代码或实现细节
- 若功能专题存在明显的成功态、进行中状态与失败反馈，验收标准应尽量覆盖这三类外显结果，而不只写理想路径
- 若专题明确会直接驱动实现，验收标准应尽量能直接映射为后续测试用例或手工验收步骤
- 若未来该能力引入 snapshot / golden 测试，`spec.md` 只描述应被验证的外显结果与差异，不写具体工具命令、基线文件名或接受流程
- `spec.md` 写需求与验收，具体测试框架、fixture、mock、snapshot 与测试实现细节留在代码仓库而不是写进 spec
- 历史规范归档在 [`_archive/`](./_archive/) 下，归档内容默认只读
