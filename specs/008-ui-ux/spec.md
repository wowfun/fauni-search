# 008 应用界面与体验 (UI/UX)

定义 FauniSearch 的全应用界面与体验语义，明确应用壳层、主工作区、非搜索管理入口、共享产品语言，以及这些体验如何承接正式公开接口族。

## 关键术语 (Terminology)

- 应用壳层（App Shell）
- 上下文条（Context Rail）
- 工作区（Workspace）
- 状态胶囊（Status Capsule）
- 搜索工作区（Search Workspace）
- 来源浏览工作区（Source Inventory Workspace）
- 设置工作区（Settings Workspace）

## 范围

- 全应用信息架构、壳层布局与工作区语义
- 应用壳层中的当前库上下文、工作区切换、状态反馈与次级入口
- 搜索、来源浏览、设置三类正式主工作区在同一壳层中的分工
- 非搜索管理体验与对应的非搜索控制面接口族
- 产品级视觉语言、文案与术语原则
- 当前阶段 UI 落地目标在 `008` 专题中的承接方式

范围外：
- 搜索请求、搜索结果、过滤分页与搜索结果交互语义
- 非搜索控制面接口的请求 / 响应形状、错误载荷与分页编码
- 任务状态机、取消/恢复内部语义与健康判定算法
- 物理 schema、目录布局、迁移版本与存储实现
- 设计 token、像素级布局、精确断点与具体组件实现
- 前端框架、状态管理库与具体路由实现

接口契约承接：
- 非搜索控制面接口族的具体请求 / 响应形状、动作回执、任务快照与健康快照编码由 [009-interfaces-and-protocol-contracts](../009-interfaces-and-protocol-contracts/spec.md) 定义

## 设计原则

- 应用优先于页面（Application Before Screens）：先定义统一壳层、上下文与工作区，再定义单页或单表单体验
- 搜索主舞台优先（Search As Default Stage）：`Search` 是默认主页与主舞台，管理能力后置但不能缺席
- 管理邻接而非压顶（Management Adjacent, Not Dominant）：来源浏览、设置、任务、诊断与维护入口不应挤占搜索首屏
- 同壳层连续性（Same-Shell Continuity）：`Search`、`Inventory`、`Settings` 必须属于同一产品壳层，而不是三套彼此割裂的工具页
- 奥卡姆剃刀（Occam's Razor for Interfaces）：每个界面块只保留直接支持当前任务的信息与控件；能由结构、上下文、选择器或状态表达的内容，不再用额外标题、说明或重复字段再次表达
- 极简工作面（Minimal Working Surface）：输入框、选择框与按钮应优先同轴、同层、紧凑排列；低频配置、技术标识与解释性文案默认后置到折叠区、详情区或专门工作面
- 共享组件一致性（Shared Component Consistency）：当前库、搜索范围、对象详情、预览、标签、按钮、列表项、notice 与 empty 等同用途界面元素必须只有一套产品语义；工作区差异只能通过共享组件的变体、插槽或密度参数表达
- 选择态层级清晰（Selected State Must Read Stronger）：tabs、toggle、segmented-like 选择器与工作区切换中的当前选中项，必须比未选中项更重、更明确；不得出现未选中项更深、当前项更透明的反直觉层级
- 当前库显式（Visible Library Context）：应用必须始终让用户知道当前操作作用于哪个库
- 阅读式结果关系（Readable Result Relationship）：结果与详情默认应维持同页阅读关系，而不是频繁跳转独立对象页
- 中文任务语言优先（Chinese Task Language First）：主界面以中文任务表达为先，工程术语后置到次级说明、折叠区或诊断面

## 应用壳层与主工作区

- 应用壳层是全局导航、当前库上下文、工作区切换、状态反馈与次级操作入口的统一承载层
- 壳层顶部必须承接稳定的上下文条，而不是首页 hero、说明横幅或开发链接墙
- 桌面端上下文条当前目标应收敛为紧凑 app bar，而不是继续扩张成多卡片 hero 区
- 桌面端上下文条当前不应同时展示工作区长摘要、等权库信息卡和独立动作带；当前生产切片优先采用“顶部轻量品牌 / 状态 + 左侧稳定导航”的软件式骨架
- 当主工作区切换已经稳定进入左侧栏时，顶部上下文条不应继续重复承载等权工作区切换或完整库 cluster
- 应用至少应存在以下正式主工作区：
  - 搜索工作区
  - 来源浏览工作区
  - 设置工作区
- `Search` 是默认主工作区
- 库创建、库切换、来源根管理、任务观察、运行时诊断、维护动作、收藏与搜索历史等能力可以进入应用壳层，但默认属于次级入口、次级流程或专门子面，不应与三大主工作区争夺首屏
- 状态摘要、任务观察与运行时诊断必须有单一正式归属；当前阶段统一收敛到 `设置 > 诊断`，不得在壳层中再扩张出第二个辅助工作面
- 应用壳层中的后台轮询、异步刷新或状态同步不得抢走当前可编辑控件的焦点，也不得清空用户尚未提交的本地输入草稿
- 应用壳层中的后台轮询、异步刷新或状态同步不得在当前详情对象未变化时周期性重建详情面中的预览控件；当前选中对象的图片、视频或文档页预览不得因为无关状态刷新而重复加载
- 应用壳层与导航的正式事实源由 [app-shell-and-navigation.md](./app-shell-and-navigation.md) 定义

## 工作区分工

- 搜索工作区承接统一搜索入口、查询模式切换、查询输入、结果浏览、详情阅读与共享状态反馈
- 来源浏览工作区承接库级来源观察、过滤、核对与来源详情浏览，不再作为搜索首屏上的内嵌大表
- 设置工作区承接 provider 配置、全局内容类型、当前库覆盖、模型测试与诊断等非搜索主编辑流
- 搜索工作区内部的共享界面与体验约束由 [search-workspace.md](./search-workspace.md) 定义
- 来源浏览工作区内部的共享界面与体验约束由 [source-inventory-workspace.md](./source-inventory-workspace.md) 定义
- 设置工作区内部的共享界面与体验约束由 [settings-workspace.md](./settings-workspace.md) 定义

## 非搜索管理体验

- 库管理流至少应覆盖：创建、删除、重命名、归档，以及从归档状态恢复到活跃状态的等价生命周期操作
- 库创建入口至少应提供：
  - 必填或等价必填的 `display_name`
  - 可选自定义 `library_id`
  - 当未显式提供 `library_id` 时，界面可只收集 `display_name`，由系统按稳定 slug 规则生成 `library_id`
- 应用界面在展示当前库、库选择器或与库相关的设置摘要时，必须以 `display_name` 作为主展示名；稳定 `library_id` 必须仍然可明确获取，但不应在同一主界面块里与相同 `display_name` 重复占据独立正文行
- 归档是可逆生命周期状态，不得等同于物理删除；已归档库必须仍然可被明确识别、查看并恢复
- 当前切片中，已归档库仍可出现在库选择器中，但必须以明显弱化或状态标签与活跃库区分
- 应用必须提供导入、刷新、重扫、重建、清理与维护动作的明确用户入口，并能向用户表达动作已进入后台执行系统
- 当前切片中：
  - `refresh` / `rescan` 继续覆盖来源管理动作
  - `rebuild` 至少应提供全库级重建入口，用于显式触发一次面向当前覆盖范围的全量重建
  - `cleanup` 至少应提供退役索引清理入口；该动作属于当前库维护动作的一部分，当前阶段应收敛到 `库管理` 中默认折叠的“库维护”区
- 当前阶段来源管理至少应同时支持库级与来源根级 `refresh` / `rescan` 入口，但这些入口默认应落在次级面或次级动作中，而不是搜索首屏主层级
- 任务观察与运行时健康当前必须统一进入 `设置 > 诊断`；来源根配置与当前库维护必须统一归属 `库管理`，而不是分散在壳层抽屉、设置桥接卡和工作区局部说明中
- 设置工作区必须承接：
  - provider 与 provider models 的 runtime overlay 编辑入口
  - 全局 `content_types` runtime overlay 的最小编辑入口
  - 当前库 `libraries.<library_id>.content_types` runtime overlay 与已解析内容模型摘要
  - 当前 provider + model 草稿测试与运行时诊断入口
- 来源浏览工作区必须作为正式管理工作区存在；搜索工作区最多保留来源摘要与跳转入口，不应继续内嵌完整来源清单

## 产品语言与当前阶段目标

- `008` 除正式工作区语义外，还需要承接产品级视觉语言、文案与术语原则；其正式事实源由 [shared-product-language.md](./shared-product-language.md) 定义
- `008` 允许存在“当前阶段 UI 落地目标”文档，用于承接不宜写成长期刚性规范的宽度、比例、响应式与组件实现目标；其当前事实源由 [current-targets.md](./current-targets.md) 定义
- `current-targets.md` 属于 `008` 专题下的实施指导文档，不替代本专题或其他专题中的长期稳定语义
- `008` 允许存在用于设计讨论与方向校准的补充原型；当前 `Search` 工作区的补充原型说明位于 [prototypes/search-ready-prototype-notes.md](./prototypes/search-ready-prototype-notes.md)
- `Ready 空态` 与 `Ready 结果态` 分别位于 [prototypes/search-ready-empty.html](./prototypes/search-ready-empty.html) 与 [prototypes/search-ready-results.html](./prototypes/search-ready-results.html)
- 这些 HTML 是补充性设计工件，用于验证搜索首页与结果态的层级、气质与信息关系；解释性说明、评审要点与非目标统一收在 `search-ready-prototype-notes.md` 中，HTML 本身应尽量接近真实 UI，而不是承载设计讲解
- 当前生产落地应承接补充原型中已经稳定的 IA：顶部只保留品牌与全局状态，左侧栏主导航承接 `Search / 库管理 / 设置`，搜索范围贴近 composer 下方表达
- 当前生产落地中的顶部品牌区当前应继续向补充原型收口：保留品牌名与极短 tagline 即可，不再重复渲染“本地多模态搜索 / 统一检索...”式标题加说明块
- 当前生产落地中，左侧栏的 `库管理` 是 `Inventory` 工作区在用户界面中的可见标签；其内部继续承接来源浏览与库级管理能力
- 当前生产落地中的 `库管理` 工作区当前应把库选择、显示名称编辑、生命周期动作与 readiness 收敛为一条低高度管理工具带；该工具带只回答“正在管理哪个库、当前是否 Ready、可执行哪些库级动作”，不再演化成二级 dashboard
- 当前生产落地中的 `库管理` 工作区当前仍要表达搜索 readiness，但主层级只保留状态标签与必要计数；来源根、watcher、待处理任务和内容类型覆盖的解释性细节留给来源列表、来源根编辑和 `设置 > 当前库覆盖`
- 当前生产落地中的 `Search` 工作区当前也应复用同一套 readiness 原因模型；搜索舞台状态条与单行 readiness 动作不应再独立发明“需要内容”式粗粒度文案，而应和 `库管理` / `设置 > 当前库覆盖` 一起回答“缺的是来源根、内容、配置还是任务完成”
- 当前生产落地中的 `Search` 工作区当前应把“去哪里解决问题”直接做成极简正式入口：来源根、导入、refresh / rescan 优先指向 `库管理`，内容类型与 resolved model 问题优先直达 `设置 > 当前库覆盖`
- 当前生产落地中的搜索空结果、失败反馈与壳层状态摘要当前也应沿用同一套 readiness 语气；这些地方必须明确告诉用户“这次没命中 / 这次失败”与“当前库不可搜”不是同一件事，并在问题属于配置或来源准备时提供直接入口
- 当前生产落地中的搜索范围条当前不应再停留在“单库范围”静态标签；第一阶段应正式支持 `当前库` 与 `所有库` 两种范围切换，并为未来 `多库` 保留明确扩展位
- 当前生产落地中的搜索范围条当前应继续向补充原型收敛成紧邻 composer 的轻量选择层：保留范围切换与当前库选择即可，`新建库 / 管理当前库` 的正式入口统一移入 `库管理` 工作区
- 当前生产落地中的文本搜索 composer 当前应进一步压缩成单行工作带：搜索图标、输入框、轻量过滤与模式入口留在同一水平层，不再额外占据下一行主动作位
- 当前生产落地中的搜索提交动作当前应从 composer 下方回收到范围条右侧，与当前库选择保持同一行；搜索进行中时，反馈当前也只保留结果区顶部的简短 `搜索中...`
- 当前生产落地中的 `Search` 工作区不再暴露 `demo import`、`快速验证` 或等价内部验证快捷入口；搜索首页只保留正式输入、正式范围与必要的单行动作提示，不再内嵌导入表单或来源根管理面板
- 当前生产落地中的 `所有库` 范围当前只要求支持文本查询；当用户切换到 `Image` / `Video` / `Document` 模式时，界面必须明确退回单库或显式表达当前范围不受支持，而不是伪装为已具备跨库非文本搜索
- 当前生产落地中的 `所有库` 结果工作面当前不应只把命中库分布做成静态摘要；用户应能留在 Search 内先按命中库聚焦结果，再决定是否进入 `库管理`
- 当前生产落地中的跨库结果当前应避免重复表达库身份；当结果已经被命中库分组或聚焦时，库身份应优先由分组标题与结果工作面摘要承担，而不是在每张结果卡上重复堆叠同一信息
- 当前生产落地中的结果 / 详情阅读面当前应继续去除已经由版面关系表达清楚的标题层；结果列优先保留命中摘要，详情列优先直接进入预览与对象信息，而不是重复渲染“搜索 / 结果 / 详情 / 结果详情”式栏位标题
- 当前生产落地中的 `document_page` 结果卡当前不应继续依赖实时 PDF iframe 缩略图；结果列表应改用轻量静态占位或等价文档卡表达，避免长 PDF 在滚动列表中频繁重载
- 当前生产落地中的轻量文档占位当前应优先表达文档类型本身，例如 `PDF`；不应在占位图里重复渲染“文档页”或具体页码，这些页级信息继续留在结果卡正文中承接
- 当前生产落地中的详情主动作当前不应与结果卡动作重复堆叠；当结果卡已经稳定提供打开预览与复用为查询输入的入口时，详情面应回到预览与结构化信息本身
- 当前生产落地中的结果缩略图区当前不应再叠加第二层对象类型文字；文档占位若已经显示 `PDF`，就不应再额外渲染一层“文档页”
- 当前生产落地中的详情标题区当前不应重复显示完整来源 URI；若路径已在结构化字段中保留，标题区只需承接文件名与必要标签
- 当前生产落地当前可以选择性吸收补充原型里已经稳定的极简英文表达，例如英文 tagline、顶层状态胶囊中的 `Ready`，以及空态搜索舞台的 `Search anything you want`；这不要求整个产品放弃中文任务语言，只要求壳层与搜索主任务表达更紧凑
- 当前生产落地中的顶栏状态胶囊在 `Ready` 状态下，当前应使用绿色文字表达 readiness 语义；不得被壳层按钮通用字色覆盖成中性墨色
- 当前生产落地中的顶部状态胶囊当前只作为全局状态入口存在；点击后应直接进入 `设置 > 诊断`，不再打开独立 drawer、popover 或第二工作面
- 当前生产落地中的全局后台任务摘要应收进顶部状态胶囊；胶囊可显示所有库 `queued` / `running` 任务数量与聚合进度条，但不得在工作区上方展开成整宽任务横幅
- 当前生产落地中的状态胶囊即使因运行时或 provider 降级显示 `部分受限`，也应在后台任务活跃时保留进度条或等价活动动效
- 当前生产落地中的 `设置` 工作区当前应取消重复的章节概览 hero、常驻指标卡、解释性面板和同义标题；左侧只负责选择章节，右侧直接进入当前章节的 runtime overlay 配置、测试或诊断工作面
- 当前生产落地中的 `设置` 工作区当前库上下文应与 `库管理` 复用同一套紧凑顶部工具栏骨架；设置页只保留当前库身份、生命周期、readiness 与切库表达，不复制重命名、归档、删除或新建库动作
- 当前生产落地中的工作区切换、Search 范围切换、Search 模式按钮、设置章节导航、内容类型 tabs、覆盖模式切换与诊断页内的可切换选择控件，当前都应复用同一套共享选择态语义：未选中保持浅纸面/描边，当前选中项使用更明确的浅白绿色底、深色文字与轻 teal 边框，不再采用高冲击的深色实底
- 当前生产落地中的 `库管理 / 设置` 当前应采用极简职责边界：
  - `库管理` 只回答当前库身份、来源根、来源列表、来源详情与当前库维护动作；来源根和维护动作都作为次级折叠区呈现，不再长成 dashboard
  - `设置` 只承接配置编辑、模型测试、运行时诊断与 Jobs 观察，不再重复展示库来源健康、维护桥接说明或章节级营销式概览
  - `设置` 中的 `libraries` 只表示当前库内容类型覆盖，不承接库生命周期或来源管理 CRUD
- 当前生产落地中的 `库管理` 工作区当前应把来源过滤条与来源根管理入口收敛为同一顶部组合组件：过滤条件、筛选摘要与唯一一组库级 `refresh / rescan` 常驻；来源根管理作为该组件内的次级折叠区，而不是过滤条外的独立常驻面板
- 当前生产落地中的来源根创建 / 编辑属于 `库管理` 的次级动作；默认浏览态只显示来源根摘要与进入编辑的入口，不应把路径、glob、扩展名等高级字段常驻在主工作面
- 当前生产落地中的来源根表单当前应把 `include_globs`、`exclude_globs` 与 `include_extensions` 收敛为“高级规则”折叠区：新建与无规则编辑默认折叠，编辑已有规则的来源根时自动展开；来源根创建 / 编辑统一归属 `库管理`
- 当前生产落地中的 `库管理` 桌面双栏当前必须优先保证来源列表里的文件名与完整路径可扫读；当两栏宽度发生权衡时，应先压缩详情预览列，而不是让来源主文本列过早折行成窄条
- 当前生产落地中的维护动作必须有单一主归属；当前库级 `refresh / rescan / rebuild / cleanup retired vector spaces` 统一归 `库管理`，其中 `rebuild / cleanup` 当前应收进默认折叠的“库维护”区
- 当前生产落地中的 `设置 > 诊断` 当前应成为唯一的运行时 / 任务观察面：运行时健康与执行空间诊断常驻显示，Jobs 作为默认折叠区呈现，并支持从状态胶囊或 Search 的单行“查看任务”入口直接展开；任务卡片应提供逐任务进度条与当前数值摘要
- 当前阶段 Search 首页应避免把搜索舞台、结果区和详情区做成首屏等权的大卡并排；无结果阶段必须以搜索舞台为唯一视觉主角，结果与详情只在形成真实阅读流后展开
- 当前阶段 Search 首页在尚未发起查询前，不应再展示“结果会出现在这里”式占位解释卡；搜索舞台本身已经足够承接空态
- 桌面端当搜索已产生稳定结果流时，搜索舞台仍应保留在结果区之上作为整宽首行；结果列表与详情面在其下形成双栏阅读面，不得把搜索舞台退回左侧三分之一栏位
- 桌面端当搜索已产生稳定结果流时，搜索舞台当前还应压缩成更薄的工作带语义；它可以继续编辑查询，但不应再像第二张厚重首屏卡压住下方阅读面
- 当前阶段壳层中的库级创建 / 管理入口若展开为 popover，必须表现为锚定在上下文 cluster 下方的浮层，不得把紧凑 app bar 重新撑成内容画布；后台轮询或同页重绘期间也不得把已展开的浮层静默关闭

## 非搜索控制面接口族

- 本专题中的控制面接口族是支撑应用体验的稳定公开入口，不是页面行为的内部实现细节描述
- `008` 至少固定以下非搜索接口族的存在与职责：
  - 库管理接口族
  - 来源根与规则管理接口族
  - 配置与绑定管理接口族
  - 任务动作接口族
  - 运行时健康与维护入口
  - 收藏 / 搜索历史管理接口族（若该辅助状态在当前产品切片中正式暴露）
- 这些接口族应覆盖应用壳层与各工作区中的正式管理动作，但不承接搜索端点
- 搜索端点的行为语义继续由 [004-search](../004-search/spec.md) 承接，不并入本专题
- 任务与运行时健康的公开入口可以出现在本专题中，但其状态、取消、恢复与健康语义本身继续复用 [006-runtime-and-execution](../006-runtime-and-execution/spec.md)
- 控制面接口族可以是 HTTP、IPC 或等价公开管理边界；其具体请求 / 响应形状、错误载荷与分页编码由 [009-interfaces-and-protocol-contracts](../009-interfaces-and-protocol-contracts/spec.md) 定义
- 当前切片中，库管理接口族除创建、重命名、删除外，还至少需要提供 archive / restore 一对可逆生命周期动作
- 当前切片中，任务动作接口族至少需要提供：
  - 面向 queued / running 后台任务的显式 cancel 入口
  - 面向 terminal failed / canceled 且被声明为 retryable 的后台任务的显式 retry 入口
- 当前切片中，任务动作接口族还至少需要提供：
  - 面向 terminal failed / canceled 且仍保留 replayable request 的后台任务的显式 resume 入口
- 当前切片中的 retry 不要求覆盖所有任务种类；至少应覆盖具有稳定重放语义的手动 import、手动来源管理与维护任务
- 当前切片中的 resume 与 retry 语义必须显式区分：
  - `retry` 重新排队一个新的 job，原 job 保留历史终态
  - `resume` 复用原 job_id 进入下一次 attempt，继续承接同一逻辑 job
