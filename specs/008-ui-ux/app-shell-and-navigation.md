# 008 应用壳层与导航 (App Shell and Navigation)

定义 FauniSearch 应用壳层、上下文条、主工作区切换、状态入口与次级辅助面的共享语义。

## 关键术语 (Terminology)

- 应用壳层（App Shell）
- 上下文条（Context Rail）
- 紧凑应用栏（Compact App Bar）
- 当前库切换器（Current Library Switcher）
- 状态胶囊（Status Capsule）
- 统一辅助面（Utility Drawer）
- 主工作区（Primary Workspace）

## 范围

- 应用壳层的正式职责与稳定组成部分
- 当前库上下文、工作区切换、状态入口与统一辅助面的共享语义
- 主工作区与次级入口之间的层级关系
- 壳层内的全局反馈与后台刷新约束

范围外：
- 具体工作区内部的布局与组件实现
- `/health`、`/jobs`、来源管理等接口的请求 / 响应编码
- 像素级尺寸、精确断点与视觉 token

## 设计原则

- 壳层先于页面（Shell Before Screens）：用户应先感知稳定产品壳层，再进入具体工作区内容
- 当前库先于动作（Library Context Before Actions）：上下文必须先于任何搜索、浏览或设置动作被用户感知
- 范围贴近任务（Scope Near Task）：当库主要扮演搜索范围时，其主表达可以靠近搜索舞台，而不必永久固定在顶部壳层
- 搜索为默认主页（Search Is Default Home）：主导航必须围绕 `Search` 默认主页组织
- 状态与工具协同（Status And Utilities With Clear Roles）：状态入口与工具入口可以共同唤起同一辅助面，但它们的触发语义与信息层级必须保持清晰
- 切换不断裂（Switch Without Context Loss）：用户在主工作区之间切换时，不应失去当前库上下文与本地草稿

## 上下文条

- 应用壳层顶部必须存在稳定的上下文条
- 桌面端上下文条当前目标应表现为紧凑 app bar，而不是品牌、库卡片、状态卡片和工具卡片并列的 hero 区
- 上下文条至少应承接以下元素：
  - 品牌
  - 状态胶囊
- 当前生产切片中，当前工作区切换入口与工具入口应优先进入稳定左侧栏；顶部上下文条不再重复承载一套等权工作区切换
- 当前库上下文可以根据任务位置进入当前活动工作区附近；当库主要表达搜索范围时，其主表达应贴近搜索舞台
- 上下文条不应再承担 hero 式说明块、开发服务裸链接或大段任务摘要

## 当前库与库入口

- 当前库切换器是上下文条中的核心元素，用于表达全局工作上下文
- 当前库切换器、活跃/归档状态、稳定 `library_id` 与库级管理动作当前可以收敛为同一个紧凑上下文 cluster
- 新建库属于首用路径关键动作，但桌面端当前可以作为当前库 cluster 中的邻接动作，不再要求独立常驻成整块按钮区
- 当前库上下文在主工作区切换中必须保持稳定可见
- 当库主要扮演搜索范围时，其主表达可以下沉到搜索舞台附近的范围条；此时顶部壳层不必重复保留完整当前库 cluster
- 当前生产切片中，Search 工作区优先采用 composer 下方范围条承接当前库表达；Inventory / Settings 则在各自工作区头部保留紧凑库上下文与管理入口
- 当用户未选定库时，应用壳层仍必须保留明显的选库或建库入口
- 当前库 cluster 中的 `新建库`、`管理当前库` 等轻量管理入口若采用 popover / disclosure 承载，其展开内容必须表现为锚定浮层，而不是把 app bar 本身撑高成第二工作面
- 当前库 cluster 中已展开的库级管理浮层在后台轮询、异步刷新或同页重绘期间必须保持打开；正在编辑的显示名称草稿不得因壳层重绘而丢失或自动折叠

## 主工作区切换

- 应用壳层必须允许用户在正式主工作区之间显式切换
- 当前阶段的主工作区至少包括：
  - `Search`
  - `Inventory`
  - `Settings`
- 主工作区切换必须发生在同一壳层内，避免每次切换都产生“换到另一套工具页”的感觉
- 当前生产切片应把主工作区切换落地为左侧栏，并以 `Search / 库管理 / 工具 / 设置` 的顺序组织入口
- 其中：
  - `Search` 对应正式 `Search` 工作区
  - `库管理` 是正式 `Inventory` 工作区的用户可见标签
  - `工具` 打开统一辅助面，不构成第四个正式主工作区
  - `设置` 对应正式 `Settings` 工作区
- 从一个主工作区切换到另一个主工作区，再切回时，本地草稿、已加载结果与详情选中态应尽可能保留

## 状态胶囊

- 状态胶囊用于表达系统当前是否可顺畅工作，例如：
  - Ready
  - 部分受限
  - 准备中
- 状态胶囊应采用仪表状态语义，而不是高噪声告警 badge
- 点击状态胶囊后可以打开统一辅助面中的状态摘要分段，而不应膨胀成独立下坠的大型 popover 工作台
- 状态胶囊可以提供进入诊断面或运行时健康面的深链接

## 统一辅助面

- 工具入口与状态入口在桌面端当前可以共同唤起同一个统一辅助面
- 统一辅助面用于承接次级操作与次级观察入口，而不是与搜索结果详情面混用
- 统一辅助面至少可以进入以下类别：
  - 系统状态摘要
  - Jobs
  - Source Roots
  - Runtime / Diagnostics
  - Maintenance actions
- 统一辅助面不应演化成 command panel、开发者控制台弹层或第二主工作区

## 全局反馈与后台刷新

- 应用壳层可以承接后台任务状态、运行时降级提醒与维护动作结果，但这些反馈不应改写底层专题中的语义
- 全局反馈应尽量采用不抢主舞台的方式呈现
- 后台轮询、异步刷新或状态同步不得抢走当前可编辑控件的焦点，也不得清空用户尚未提交的本地输入草稿
- 后台轮询、异步刷新或状态同步不得在当前详情对象未变化时周期性重建预览控件
