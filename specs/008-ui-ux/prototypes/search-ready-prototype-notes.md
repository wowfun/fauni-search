# Search Ready Prototype Notes

`Search` 工作区的补充原型拆成两个独立 HTML：

- [Ready 空态](./search-ready-empty.html)
- [Ready 结果态](./search-ready-results.html)

## 用途

- 用于讨论 `Search` 首页与结果阅读态的层级关系
- 用于比较“搜索前”和“进入阅读流后”是否已经是两种明确不同的产品节奏
- 用于校准左侧栏、搜索范围条、搜索舞台、结果列表和详情面的相对主次

## 本轮调整

- 不再用单页状态切换器承载两个状态
- 顶部完整当前库 cluster 与顶部工作区切换从原型中移除
- 顶部只保留品牌与全局状态
- 左侧栏改为 `Search / 库管理 / 工具 / 设置`
- 搜索框右侧加入 `过滤` 按钮和 `图片 / 视频 / 文档` icon-only 入口
- 库上下文改为 composer 下方的搜索范围条
- 品牌副标题、状态词与空态主标题允许切到英文表达，以验证产品语气
- `最近查询` 需要支持文本、图片、视频、文档四类历史查询的显示形态
- `最近对象` 改为“类型标签 + 文件名 + 单行路径”的更紧凑对象卡
- 所有设计说明、原型注记和评审语句继续留在本文件；HTML 只保留应当像真实产品一样出现在界面里的元素与文案

## 两个页面分别验证什么

### Ready 空态

- 软件式壳层是否成立：顶部轻、左栏稳、主舞台集中
- 搜索入口是否是唯一主角
- 搜索范围条是否比顶部库块更接近真正的查询语义
- `最近查询` 与 `最近对象` 是否只是辅助记忆，不抢搜索舞台

### Ready 结果态

- 搜索舞台是否压缩成工作带
- `Results / Detail` 是否形成明确双栏阅读关系
- 结果卡是否足够适合扫读
- 详情面是否更像阅读面而不是工具侧栏
- 结果态是否仍然沿用同一左侧栏与同一范围表达，不重新回到顶部导航

## 明确不做

- 不接入真实 API
- 不接入 Vite 或生产路由
- 不展开库管理、工具或设置工作区的完整交互
- 不把范围条做成真实多库切换器
- 不覆盖 `无库 / 无内容 / not_ready`

## 评审建议

- 桌面宽度先看 `1440px` 左右
- 再看 `375px` 是否还能保持清晰堆叠
- 重点判断 HTML 本身是否已经像真实 UI，而不是靠解释性文字告诉评审“这里想表达什么”
- 重点观察库是否已经被用户感知为“搜索范围”，而不是壳层里的一块全局卡片
