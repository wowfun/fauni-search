# 008 搜索工作区 (Search Workspace)

定义 FauniSearch 搜索工作区的共享界面与体验约束，明确不同查询能力专题如何在同一搜索工作区中进入、表达前置状态、展示结果并展开详情。

## 关键术语 (Terminology)

- 搜索工作区（Search Workspace）
- 搜索舞台（Search Stage）
- 查询模式（Query Mode）
- 查询输入卡（Composer）
- 下一步提示区（Next-Step Dock）
- 结果列表（Result List）
- 详情面板（Detail Panel）
- 状态条（State Strip）

## 范围

- 搜索工作区内部的共享布局与交互骨架
- 查询模式切换、搜索舞台、共享搜索入口与非文本查询模式的共享交互约束
- 首用 / 未就绪 / 空结果 / 失败等共享状态在搜索舞台中的表达方式
- 结果列表、详情面板、对象复用与共享状态反馈的工作区语义
- 搜索工作区与当前库上下文、来源浏览工作区、设置工作区的关系

范围外：
- 具体查询能力的输入语义、结果对象边界与当前阶段范围
- `/search/*`、详情接口或上传接口的请求 / 响应编码
- 搜索排序、过滤、分页、调试载荷与错误码触发条件
- 应用壳层、上下文条与全局导航关系
- 设计 token、像素级布局与具体组件实现

## 设计原则

- 搜索首页优先（Search As Home）：`Search` 是默认主页，用户进入应用后应首先看到搜索主舞台而不是管理大表或运维面板
- 工作区连续性（Workspace Continuity）：查询输入、结果浏览与详情浏览应在同一工作区内连续完成，而不是频繁跳转独立页面
- 当前库上下文显式（Visible Library Context）：搜索工作区必须始终让用户知道当前查询作用于哪个库
- 范围贴近查询（Scope Near Query）：当库主要表达搜索范围时，应优先贴近 composer 表达，而不是远离主输入
- 搜索主路径优先（Search Above Management）：搜索主路径应占据首屏重心；导入、来源浏览、任务观察与维护入口属于邻接路径
- 统一入口、单次单输入（Unified Entry, Single Query Input）：工作区可以承接统一搜索入口与查询模式切换，但单次公开搜索请求只应发出一种查询输入
- 非文本模式可复用（Reusable Non-Text Modes）：`Image`、`Video`、`Document` 等非文本模式应共享稳定的查询输入交互，而不是各自发明独立工作区
- 阅读式结果关系（Readable Result + Detail）：结果列表与详情面板应构成阅读式关系，而不是“点开后跳转对象页”
- 共享状态明确（Explicit Shared States）：无库、无内容、未就绪、空结果与失败必须在工作区中明确区分，不得互相伪装

## 搜索主舞台

- 搜索工作区必须把搜索舞台作为首屏主体区域
- 桌面端当结果尚未形成稳定阅读流时，搜索舞台应独占主要宽度，而不是与空的结果列或空的详情列并列占位
- 搜索舞台至少应包含以下稳定层次：
  - 当前查询输入卡
  - 搜索范围条
  - 主搜索动作与轻量过滤入口层
  - 状态 / 帮助条
- 当前生产切片中，空态搜索舞台头部应进一步收敛为单一主任务语句；不再额外叠加 `搜索 / 开始搜索` 这类重复 eyebrow + 标题组合
- 当前生产切片中，轻量过滤入口与 `Image / Video / Document` 等非文本模式入口可以直接并入 composer 右侧；当 `Text` 是默认模式时，不再要求等权文字 tabs 占据首层
- 搜索舞台必须始终可见；即使在“无库”“有库但无内容”或“未就绪”状态下，也不应退化成完全独立的创建页或错误页
- 搜索舞台可以承接轻量导入、下一步提示或对象复用入口，但这些动作不应与主搜索动作争夺同层主视觉
- 当前生产切片中，搜索舞台与下一步提示区不再暴露 demo fixture、内部快捷验证或内部 smoke 入口；正式用户路径只保留真实查询、真实导入与真实管理动作
- 搜索舞台顶部说明当前应保持极简，只承接当前任务标题与一行必要帮助语，不应再次长成 marketing / onboarding 风格的说明块
- 桌面端当结果列表与详情面已经出现时，搜索舞台仍应作为它们的上层整宽行存在；不得把搜索舞台、结果列表、详情面重新排成三个直接并列的大栏
- 桌面端当结果列表与详情面已经出现时，搜索舞台应进一步压缩垂直高度与装饰强度，更接近工作带或控制条，而不是第二张与下层阅读面等权的主卡

## 查询输入与模式切换

- 搜索工作区可以提供统一搜索入口，用于承接不同查询能力专题的查询模式
- 查询模式切换属于工作区体验的一部分，但具体有哪些正式查询模式由各搜索能力专题定义
- 非文本查询模式必须在搜索舞台中提供稳定的查询输入卡，而不是把上传、对象复用、预览或替换动作散落为隐式旁路
- 查询输入卡必须能够表达当前输入来源状态，例如：
  - 当前使用的是文本查询
  - 当前使用的是临时上传输入
  - 当前使用的是库内对象引用
  - 当前尚未提供有效查询输入
- 查询输入卡必须支持以下共享交互能力：
  - 当前输入预览
  - 清除或替换当前输入
  - 在能力专题允许时，将库内对象复用为新的查询输入
- 当前查询输入卡可以把轻量过滤按钮与 `Image / Video / Document` 等非文本模式入口收敛在输入尾部；当 `Text` 是默认模式时，它可以不再以等权 tab 常驻占位
- 当前生产切片中的文本 composer 应压缩为单行输入带：搜索图标与输入框在同一行，轻量过滤与模式入口留在输入右侧，不再把主提交动作另起一整行
- 当前生产切片中的空态主任务标题应作为搜索舞台的中心标题出现，例如 `Search anything you want`；不再左对齐地重复一个额外的工作区标题
- 搜索范围条当前可以紧邻 composer 下方表达，例如：当前范围是单库、多库或所有库；其具体切换语义继续由工作区与相关专题共同承接
- 当前第一阶段生产切片中，搜索范围条至少应显式支持：
  - `当前库`
  - `所有库`
- 当前生产切片中，搜索范围条应收敛成轻量选择层：范围切换与当前库选择保留在搜索舞台中，库创建与当前库管理入口统一移入 `库管理` 工作区
- 当前生产切片中，搜索范围条的主行应保持单轴排版：`当前库 / 所有库`、当前库选择器与正式搜索提交按钮留在同一水平线上；若存在过滤摘要，它应退到次行而不是把主动作挤下去
- 当前生产切片中，范围条右侧应并排承接当前库选择与正式搜索提交按钮；搜索进行中时，简短状态提示应出现在结果区顶部，而不是继续占据舞台内部状态条
- 当前第一阶段生产切片中，`所有库` 当前只要求支持文本查询；当用户切换到 `Image` / `Video` / `Document` 时，工作区必须显式限制该范围，而不是继续允许用户提交一个当前后端并不支持的请求
- 当当前范围为 `所有库` 时，搜索工作区中的提交可用性、状态条、下一步提示与空结果文案都必须以“当前范围是否已有可搜索库”为准，而不是继续偷偷按当前选中库的 readiness 退化判断
- `Document` 等需要范围控制的非文本模式，可以在共享输入卡内提供页范围、时间范围等专题专属控件；这些控件的具体语义仍由对应能力专题定义
- 搜索舞台中的轻量过滤入口只承接已经进入正式搜索契约的共享控制项；更多过滤器可以进入折叠式高级区，但不应把搜索首屏变成长表单
- 非文本查询模式当前应共享同一套材料层级：输入来源 / 状态摘要、当前输入预览面、范围或专题控件、清除 / 替换动作；不应让 `Image`、`Video`、`Document` 三种模式继续表现为三套互不相干的卡片语法

## 下一步提示区与轻量导入

- 搜索舞台可以承接“下一步提示区（Next-Step Dock）”或等价结构，用于表达用户当前最合理的下一步动作
- 下一步提示区应优先用于：
  - 无库时的建库或选库引导
  - 尚未接入来源根、来源根已停用或观察未稳定时的来源准备 / 库管理引导
  - 来源根已接入但仍无可搜索对象时的导入 / refresh / rescan 引导
  - 当前库覆盖或 resolved model 未就绪时的配置引导
  - 后台任务仍在准备当前库时的任务观察引导
  - 必要时的轻量导入入口
- 下一步提示区只在需要时出现，不应长期占据与主查询输入等权的视觉层级
- 导入入口可以出现在搜索工作区中，但应作为搜索主路径的邻接动作，而不是再次把搜索工作区变成来源管理页
- 搜索舞台中承接导入、来源准备或其他邻接操作的 disclosure / dock，在用户已经主动展开并开始编辑时，必须在轮询、任务状态刷新或同页重绘期间保持展开，不得把正在编辑的辅助面静默折叠回去
- 下一步提示区当前应表现为邻接提示卡，而不是第二主舞台；其材质、对比和装饰层级都应弱于主 composer
- 搜索舞台的状态条与下一步提示区当前应优先复用 `库管理` 工作区已经使用的 readiness 原因模型；当问题属于来源根、watcher、待处理任务或内容类型 / resolved model 时，不应在 Search 里重新发明另一套状态名
- 下一步提示区中的正式入口当前应直接把用户送到能解决问题的地方，而不是停留在泛化说明：
  - 来源根与 refresh / rescan 问题优先指向 `库管理` 或搜索舞台中的来源准备
  - 内容类型绑定、当前库覆盖与 resolved model 问题优先直达 `设置 > 当前库覆盖`
  - 任务准备问题优先指向任务面或搜索舞台中的任务 disclosure

## 共享状态场景

- 搜索工作区必须显式区分以下共享状态：
  - 无库
  - 有库但无内容
  - 目标索引或运行时未就绪
  - 空结果
  - 查询或上传失败
- 无库时：
  - 搜索舞台仍然存在
  - 查询输入卡应表现为受限状态，而不是直接消失
  - 工作区必须提供建库或选库的清晰下一步动作
- 有库但无内容时：
  - 搜索舞台仍然存在
  - 查询输入卡可以继续受限，但不得与“无库”状态完全同形
  - 工作区必须进一步区分：
    - 尚未接入来源根
    - 来源根已接入但仍无内容
    - 当前库覆盖或 resolved model 仍未就绪
    - 后台任务仍在准备当前库
  - 工作区必须针对这些原因分别提供清晰下一步动作
- 未就绪时：
  - 搜索按钮可以保留但禁用，并明确表达原因
  - 工作区可以展示轻量任务摘要，但不应在搜索舞台内部膨胀成完整任务中心
- 空结果时：
  - 工作区必须表达明确空状态，而不是将空结果与未就绪或失败混淆
  - 当前库若仍处于可搜索状态，空结果文案必须明确表达“本次查询未命中”，而不是暗示当前库不可搜
  - 当空结果同时伴随被跳过的内容类型时，工作区可以提示用户检查当前库覆盖或 resolved model，但这类提示不得把空结果伪装成失败
- 尚未发起查询时：
  - 工作区不再需要额外展示“结果会出现在这里”式占位说明
  - 空态应把视觉重心留给 composer、本地范围选择与必要的下一步动作
- 失败时：
  - 工作区必须向用户表达明确失败反馈，不得静默退化为无结果
  - 失败反馈当前应尽量区分查询输入问题、运行时问题与内容类型 / 当前库覆盖问题；当失败与配置或准备状态直接相关时，工作区应提供进入 `库管理` 或 `设置 > 当前库覆盖` 的直接入口

## 结果与详情关系

- 结果列表默认应呈现可扫读的最小结果卡片，而不是高密度数据表
- 宽屏下，结果列与详情列应按状态出现；在无结果、无选择或仅首用引导阶段，不应保留结构性空白栏位
- 宽屏下，一旦结果列与详情列同时出现，它们应位于同一阅读行并共享下层工作面；搜索舞台保留在其上方，而不是成为并列的第三主栏
- 宽屏下形成阅读流后，下层结果 / 详情工作面的视觉重量应明显高于上层搜索舞台，避免用户把视线重新拉回首行大卡
- 结果列表必须支持在同一工作区中选中对象并展开详情
- 搜索成功后，工作区可以默认选中第一条结果，以保证详情面立即可读
- 结果卡与详情面都可以承接“复用为下一次查询输入”的动作；能力专题允许时，该动作应成为正式一等操作，而不是隐藏调试入口
- 当搜索范围允许跨库命中时，结果与详情中的对象动作必须尊重命中库身份；对象详情、打开预览与“复用为查询输入”都不得再隐式假设命中对象来自当前选中库
- 当搜索范围允许跨库命中时，结果工作面还应显式表达命中库分布，而不是只把库身份压缩成每张结果卡上的孤立 pill；详情面也应提供继续进入命中库上下文的正式入口
- 当搜索范围允许跨库命中时，命中库分布不应只是静态说明；工作区应允许用户在 Search 内按命中库聚焦当前结果集，并在聚焦变化时保持详情阅读连续
- 当搜索范围允许跨库命中且用户尚未聚焦单一命中库时，结果列表可以按命中库形成阅读分组；这类分组不应打断原有结果顺序所传达的优先级
- 当结果列表按命中库形成分组时，每个分组标题应成为可读的 section anchor：至少应包含命中库身份、结果数量，以及留在 Search 内继续聚焦或进入命中库上下文的正式动作
- 当命中库身份已经被结果工作面摘要、命中库分布或分组标题稳定表达时，结果卡内部不应再重复显示同一库身份标签；结果卡应回到对象、页码 / 时间片与相似度等对象级信息
- 当结果卡回到对象级信息后，其信息顺序应优先服务扫读：对象类型 / 页码或时间片 / 相似度优先于文件名出现，文件名优先于完整路径；完整路径属于次级说明
- 当结果工作面在 grouped 与 focused / 单库阅读态之间切换时，视觉层级也应同步切换：grouped 态应由分组标题主导、结果卡更安静；focused / 单库阅读态可以给予结果卡更强一点的阅读重量
- 当搜索范围允许跨库命中且用户触发“复用为查询输入”时，工作区必须先切回命中库，再进入库绑定的查询模式；不得把用户留在 `所有库` 范围里继续伪装为单库查询
- 当结果列与详情列已经由位置关系表达分工时，工作区不应再重复渲染高噪声栏位标题；结果列优先保留命中摘要，详情列优先直接进入当前对象的预览与元信息
- `document_page` 结果卡当前不应继续在结果列表中挂载实时 PDF iframe 预览；结果列表里的文档结果应优先使用轻量静态占位，以避免长文档滚动时的黑屏与重载噪声
- 详情面板同时承接可视预览与结构化信息
- 详情面必须优先承接用户可读的信息与主动作；技术字段、调试摘要与低频元数据应折叠后置
- 当结果卡已经稳定承接对象级主动作时，详情面不应再重复堆叠同一组“打开预览 / 复用为查询输入”按钮；详情面优先服务阅读，而不是成为第二个操作条
- 当当前详情对象及其预览资源引用未变化时，后台轮询或异步状态刷新不得重建详情面中的预览子树；已打开的图片、视频或文档页预览必须保持原位挂载，而不是被周期性重新加载
- 结果列表与详情面板的具体字段由各搜索能力专题继续复用 [004-search](../004-search/spec.md) 与 [009-interfaces-and-protocol-contracts](../009-interfaces-and-protocol-contracts/spec.md) 的事实源

## 响应式连续性

- 宽屏下，结果与详情应优先保持并列阅读关系
- 较窄屏幕下，结果与详情可以堆叠、切换或以抽屉 / sheet 等等价方式承载，但必须保持在同一搜索工作区中的连续体验
- 精确宽度、比例、堆叠时机与移动端详情承载方式不在本文件中硬编码；当前阶段目标由 [current-targets.md](./current-targets.md) 承接
