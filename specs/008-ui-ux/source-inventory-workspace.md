# 008 来源浏览工作区 (Source Inventory Workspace)

定义 FauniSearch 来源浏览工作区的共享界面与体验约束，明确库级来源观察如何作为正式管理工作区存在，并与搜索工作区形成清晰分工。

## 关键术语 (Terminology)

- 来源浏览工作区（Source Inventory Workspace）
- 来源过滤条（Inventory Filter Bar）
- 来源列表（Inventory List）
- 来源详情面（Inventory Detail Surface）
- 工作区切换器（Workspace Switcher）

## 范围

- 来源浏览工作区内部的共享布局与交互骨架
- 库级来源过滤、列表浏览、状态摘要与来源详情的共享观察语义
- 来源浏览工作区与搜索工作区之间的切换关系

范围外：
- 来源根生命周期、`refresh` / `rescan` / watcher 的底层语义
- `/libraries/:id/sources` 等接口的请求 / 响应编码
- 来源修复、人工纠错或 source-level detail 的独立工作流
- 设计 token、像素级布局与具体组件实现

## 设计原则

- 管理面独立（Management Surface Is Separate）：来源浏览属于正式管理 / 观察面，不应继续占据搜索入口上方的主流位置
- 当前库显式（Visible Library Scope）：来源浏览工作区必须始终保持当前库上下文可见
- 浏览器而非后台表（Browser, Not Admin Grid）：工作区的核心任务是浏览、核对与理解来源状态，而不是堆满运维表格
- 过滤可持续（Persistent Filtering）：来源根、来源类型与来源状态过滤应在工作区内保持稳定，不因切换搜索模式而被重写
- 搜索草稿不受扰（Search Drafts Stay Intact）：用户在搜索工作区中的查询草稿、结果与详情选中态，不得因为切换到来源浏览工作区而丢失
- 阅读式详情延续（Readable Detail Continuity）：来源详情应与搜索详情共享阅读式产品语法，而不是退化为开发者侧栏

## 工作区骨架

- 来源浏览工作区与搜索工作区共享同一应用壳层、当前库上下文与全局反馈区域
- 来源浏览工作区默认浏览态至少应呈现：
  - 当前库上下文
  - 紧凑库管理带
  - 来源过滤条
  - 中密度来源列表
  - 预览优先的来源详情面
- 来源浏览工作区可以显示来源摘要，但摘要只能服务浏览入口，不应把库管理带扩张成新的 dashboard
- 来源浏览工作区中的当前库管理带当前承接库选择、显示名称编辑、生命周期动作和最小 readiness 表达；它不应同时常驻来源根健康列表、内容类型配置桥接块、解释性 readiness 卡和多张指标卡
- 来源浏览工作区中的 `库操作` 与当前库显示不应作为左右两个并列区域重复出现；当前切片应压缩为同一条顶部管理工具带，当前库由选择器表达，生命周期与 readiness 以状态标签跟随显示
- 来源过滤条与来源根管理入口当前应收敛为同一个顶部组合组件；过滤条件、筛选摘要与库级次级动作常驻，来源根管理作为同组件内的次级折叠区呈现
- 来源根创建 / 编辑属于库管理的次级动作；默认浏览态只保留来源根摘要、刷新 / 重扫入口和进入来源根管理的入口
- 当当前库级内容类型覆盖或 resolved model 需要关注时，库管理带可以保留一个进入 `Settings > 当前库覆盖` 的直接入口；完整模型绑定列表不应常驻在库管理首页
- 来源浏览工作区可以在过滤区附近承接刷新当前库或重扫当前库；这些动作必须弱于浏览主路径
- 同一个来源浏览工作面中，库级 `refresh` / `rescan` 入口只应出现一组；来源根面板、过滤区与其他次级区块不得重复常驻第二组等价按钮
- 来源浏览工作区当前还应提供默认折叠的“库维护”区，承接 `rebuild` 与 `cleanup retired vector spaces`；这些低频高风险动作不应常驻在过滤主行，也不应漂移到设置诊断面

## 过滤、列表与详情

- 来源过滤条至少应支持：按来源根、来源类型与来源状态进行过滤
- 来源过滤条可以采用粘性区块或等价承载方式，但不应遮挡列表内容
- 来源列表应优先呈现中密度、可扫读的信息，而不是高密度数据表
- 来源列表当前可以为每条来源增加代表性缩略图或等价前导材质，帮助用户快速判断“这是一张图、一段视频还是一个文档来源”
- 来源列表至少应覆盖：
  - `source_uri`
  - `source_root_label`
  - `source_type`
  - `kind`
  - `status`
  - `asset_count`
  - `status_reason`（若存在）
- 来源列表当前可以把文件名与完整路径拆开表达：文件名用于首行扫读，完整路径与来源根标签退到次行或元信息行
- 桌面端来源列表中的文件名与完整路径必须获得主文本优先级；详情列、状态列或对象计数不得把主文本压缩成过窄的折行宽度
- 来源详情面应优先承接预览与用户可读信息；低频技术字段可以后置
- 来源列表与详情面不应重复堆叠同一类型事实；若缩略图、文件名或详情字段已经表达了对象类型，列表右侧与详情标签区不应再次显示 raw `kind` / `source_type` / 代表对象标签，详情标签也不应重复列表中已经承担的来源状态；普通 `正常` 状态不应在每条来源中常驻重复显示，只在异常、待处理或存在 `status_reason` 时显式呈现状态
- 来源详情面当前可以先给出当前状态、来源类型、可搜索对象数与代表性对象摘要，再把库归属、状态原因或技术背景折叠或后置；这些字段应避免和详情顶部标签、下方结构化字段重复
- 来源详情面中的图片、视频与 PDF 预览在当前来源未变化时必须跨轮询保持同一个已连接的 DOM 节点；后台刷新可以更新周边文字与列表，但不得把 iframe / video / image 移出 DOM 后再插回，避免预览控件频繁重载造成闪烁
- 当前库管理带当前可以附带来源根、可搜索对象和待处理任务的 inline 计数；完整来源根列表、规则编辑和 watcher 解释只出现在来源准备 / 来源根编辑区域
- 来源根表单中的 `include_globs`、`exclude_globs` 与 `include_extensions` 当前应作为高级规则折叠区表达；新建来源根或编辑无规则来源根时默认折叠，编辑已有规则来源根时默认展开
- 当当前筛选条件下无来源时，工作区必须显示明确空状态，而不是退化为无反馈空白

## 次级动作

- `refresh` / `rescan` 属于正式动作，但在来源浏览工作区中应保持次级层级
- 来源浏览工作区可以提供进入来源根配置、刷新、重扫、重建或清理流程的入口，但这些动作不应取代浏览主路径；其中 `refresh / rescan` 常驻于顶部动作带，`rebuild / cleanup` 收进折叠维护区

## 与搜索工作区的关系

- 工作区切换器必须允许用户在 `Search` 与 `Inventory` 之间显式切换，而无需离开当前应用壳层
- 当前生产切片中，`Inventory` 在左侧栏中的可见标签可以表现为 `库管理`，用于强调它既承接来源浏览，也承接库级管理入口
- 搜索工作区可以保留来源摘要与进入来源浏览工作区的入口，但不应继续内嵌完整来源清单列表
- 从搜索工作区切换到来源浏览工作区，再切回搜索工作区时，查询草稿、结果列表与详情选中态必须保持
