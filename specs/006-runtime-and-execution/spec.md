# 006 运行时与执行 (Runtime and Execution)

定义 FauniSearch 的运行期执行语义，明确任务如何被触发、排队、推进、恢复、取消与清理，以及本地运行时进程、健康状态、模型驻留与维护任务如何在系统内被托管与观察。

## 关键术语 (Terminology)

- 执行触发（Execution Trigger）
- 任务（Job）
- 任务尝试（Job Attempt）
- 执行检查点（Execution Checkpoint）
- 取消（Cancellation）
- 进度快照（Progress Snapshot）
- 运行时进程（Runtime Process）
- 健康快照（Health Snapshot）
- 驻留状态（Residency State）
- 清理运行（Cleanup Run）

## 范围

- 运行期执行系统、任务模型与稳定状态流转
- 触发、调度、恢复、重试、取消与进度语义
- 本地运行时托管、健康检查、失联处理与生命周期语义
- 模型驻留、设备占用、容量压力、逐出与维护 / 清理任务的执行语义
- 最小可观察面上的任务与运行时状态语义

范围外：
- 索引线生命周期、激活规则与内容版本语义
- 提供方能力声明、绑定解析顺序与探测判定规则
- 对外搜索接口、控制面接口与结果排序语义
- SQLite/Qdrant/文件系统的物理 schema 与完整 API / protocol wire shape
- worker 数量、线程模型与具体内部并发实现

## 设计原则

- 单一执行编排中心（Single Execution Coordinator）：Rust 主服务是唯一执行编排中心，负责任务创建、排队、推进、取消、恢复与本地运行时托管
- 持久执行优先（Durable Execution First）：可能跨越单次请求生命周期的执行工作，应以可恢复的持久任务表示，而不是只存在于进程内瞬时状态
- 前台搜索、后台摄取与维护（Foreground Search, Background Ingestion and Maintenance）：交互搜索保持前台请求路径；导入、来源根扫描、索引、刷新、重扫、清理、探测与维护类工作进入后台执行系统
- 本地托管显式（Explicit Local Hosting）：本地 Python sidecar 与本地检索后端由 Rust 主服务托管；远端提供方不由应用托管
- 最小可观察性承诺（Minimal Observability Guarantee）：任务与运行时至少应稳定暴露状态、进度、取消能力、健康摘要与诊断摘要

## 执行编排中心与运行时归属

- Rust 主服务是系统内唯一正式执行编排节点，负责统一承接执行触发、任务调度、状态推进与恢复逻辑
- Rust 主服务托管本地 Python sidecar 与本地检索后端进程，并负责其启动、健康检查、失联处理、重连或重启，以及关闭
- `remote_http` 一类远端提供方只参与连接、能力校验与健康探测，不由应用负责进程托管
- Python sidecar 不应成为前端/调用方的直接服务入口，也不应独立承担系统级任务调度职责
- 交互搜索请求保持前台请求路径，不被纳入后台任务系统

## 执行触发与任务范围

- 手动触发、刷新策略触发、恢复触发、清理触发、探测触发与维护触发，都是正式执行触发
- 后台执行系统至少承接以下任务种类：
  - 导入 / 来源根扫描
  - 索引
  - 刷新 / 重扫
  - 清理
  - 探测
  - 维护
- 进入后台执行系统的工作，必须经过任务创建、资源判定、排队、运行、完成/失败/取消的稳定状态流转
- 进入后台执行系统的动作型请求，在任务创建与初始任务快照生成后即可返回；后续排队、运行、完成或失败应由后台执行系统异步推进，而不是被单次 HTTP 请求生命周期阻塞
- 交互搜索不进入后台任务系统；重型搜索如何异步化不在本专题中固定

## 任务模型、检查点与状态流转

- 任务是后台执行的稳定承载单元，至少包含：任务种类、所有者作用域、阶段状态、当前任务尝试、执行检查点引用、可取消性与进度快照
- 父任务 / 子任务是正式任务关系，用于表达上层工作与阶段性子工作之间的归属与推进关系
- 任务尝试记录单次执行尝试，至少包含：attempt 序号、开始状态、结束状态、失败原因与重试归因
- 执行检查点记录可恢复的阶段边界；恢复必须从最近一次已落盘检查点继续，而不是默认从头执行
- 任务的稳定终态至少包括：成功、失败、取消
- 当单次后台任务同时推进多个 `vector_space` 时，失败终态不等同于“无任何结果生效”：若部分空间已成功完成安全边界并持久化，它们可以在任务失败终态下继续保留；失败只表示该次任务未把全部目标空间一致推进到成功
- 失败后的自动重试应是有限的；超过自动重试范围后，应转入显式失败或等待人工恢复
- 取消采用协作式语义：任务可以在安全边界停止，但不承诺在任意内部步骤立刻中断
- 资源重任务与普通任务必须具备稳定隔离语义，以避免高负载执行隐式饿死其他工作

## 调度、恢复与进度

- 调度系统至少要考虑任务种类、资源约束、当前运行时可用性与任务间隔离要求
- 刷新策略触发的摄取或索引工作进入后台执行系统后，其排队、取消、恢复与进度语义由本专题定义，而不是由 [003-ingestion-and-indexing](../003-ingestion-and-indexing/spec.md) 直接承接
- 由失败、中断或重启引发的恢复，应通过持久任务、任务尝试与执行检查点共同完成
- 当前切片中的手动 retry 可以采用“为原任务显式重放一个新任务”的形式；原任务保留其历史终态，新任务承接后续排队、运行与终态
- 当前切片中的手动 retry 不要求覆盖所有任务种类；至少应覆盖具有稳定重放语义的手动 import、手动来源管理与维护任务
- 当前切片中的显式 retry 即使采用“新任务”形式，也不应丢失 attempt 语义：新任务至少应带有递增后的 attempt 序号，并能追溯到其 `retried_from_job_id`
- 当前切片中的显式 resume 可以采用“重开同一逻辑 job”的形式：复用原 `job_id`，递增 `current_attempt.attempt`，并重新进入 queued/running 路径
- 当前切片中的显式 resume 当前只要求覆盖仍保留 replayable request 的 terminal `failed` / `canceled` 任务；若任务已保留 durable 部分成果，这些成果应继续作为 resume 的基线，而不是在 resume 前先被回滚
- 对多 `vector_space` 任务，恢复与重试应面向仍未成功的空间继续推进，而不是默认撤销已成功空间后整体重跑
- 进度快照至少应能表达：当前阶段、阶段内进度摘要、总体推进状态与是否可取消
- 本专题只固定进度语义，不固定前端展示模型或具体百分比计算算法
- 当前 restart-persistence 切片是显式例外：`jobs`、任务尝试与检查点不进入 restart-durable subset，应用重启后 `/jobs` 为空是预期行为，而不是恢复失败
- 因应用重启而中断的导入、`refresh` 或 `rescan`，当前切片不要求自动恢复执行；需要由后续显式触发重新排队

## 运行时生命周期与健康

- 运行时进程是被执行系统观察与托管的正式运行时对象，至少包括：本地 Python sidecar 与本地检索后端
- 运行时进程必须具备稳定生命周期语义，包括：启动/连接、就绪、降级、失联、重连或重启、关闭
- 健康快照至少应包含：可用、降级或不可用状态，最近一次探测结果与诊断摘要
- 本地运行时失联时，执行系统应显式进入降级或恢复路径，而不是静默忽略
- 远端提供方虽然不由应用托管，但其连通性与健康结果仍应以健康快照方式进入系统观察面

## 驻留、容量与维护执行

- 驻留状态承载已加载模型或其他运行时资源、设备占用与容量状态的稳定语义
- 本地模型可以按需懒加载并在运行期驻留；是否驻留以及何时逐出，由容量压力、健康状态与执行系统判定共同决定
- 设备占用与容量压力必须进入正式执行语义，以支持任务调度、资源隔离与运行时逐出
- 逐出与重载是正式运行期行为：被逐出的模型或资源可以在后续任务需要时重新加载
- 延迟清理、派生资产清理、临时资产回收与维护探测都由后台执行系统承接
- `vector_space` 退役后的命名空间清理也属于正式 cleanup run：后台维护循环应只在延迟窗口到期后尝试回收这些 retired 命名空间；若清理失败，应保留可重试的 retired inventory，而不是静默丢失这类待清理事实
- [002-state-and-data-model](../002-state-and-data-model/spec.md) 与 [003-ingestion-and-indexing](../003-ingestion-and-indexing/spec.md) 继续定义保留窗口、状态归属与可恢复边界；本专题只定义这些清理与维护工作如何被触发、排队、取消、重试与隔离

## 最小可观察面

- 对外至少应存在任务句柄、任务状态、阶段进度、取消能力、运行时健康摘要与诊断摘要这些稳定语义
- 任务状态与阶段进度必须能够在动作回执返回后继续被观察，例如通过任务详情或等价轮询入口看到从 queued/running 到终态的推进
- 任务观察面至少要能表达：任务是谁、在做什么、做到哪一阶段、当前是否可取消、是否处于恢复或重试过程中
- 运行时观察面至少要能表达：当前运行时是否可用、是否降级、最近一次探测结论，以及是否影响后台执行
- 当前切片中，最小运行时观察面至少应汇总：
  - app 自身健康摘要
  - 本地检索后端健康摘要
  - provider 级健康摘要
  - 本地 runtime 的工程增强 adapter 摘要
- 工程增强 adapter 摘要只属于运行时诊断面；它不得回流成 provider / model 的原生能力事实
- 本专题不固定完整公开接口或 sidecar 协议形状，但这些最小观察语义不得缺失

- 任务中心、运行时健康界面与相关控制面入口由 [008-ui-ux](../008-ui-ux/spec.md) 定义；本专题继续只承接底层执行与观察语义
